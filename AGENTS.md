# プロジェクト運用

- このプロジェクトで作業する前に、同じフォルダの `共有仕様書.md` を読む。
- 自動読み込みだけに依存せず、各作業の指示でも `AGENTS.md` と `共有仕様書.md` の確認を明記する。
- 仕様の正本は `共有仕様書.md` とし、会話履歴だけを根拠に実装しない。
- WorkまたはCodexで仕様が決まったら、実装前に `共有仕様書.md` を更新する。
- 軽微な修正も `共有仕様書.md` の変更履歴へ記録する。
- 検討中の案は未決事項へ記録し、決定済み仕様と区別する。
- 作業開始時に、プロジェクト名、保存場所、Gitブランチ、未保存の変更を確認する。
- WorkとCodexで同じファイルを同時に変更しない。
- 既存の未保存変更やユーザー作成ファイルを、許可なく削除・上書きしない。
- 実装後は必要なテストを行い、仕様書の現在の状態と変更履歴を更新する。
- commitには今回と無関係な変更を混ぜない。
- pushと公開が必要な場合は、公開先での確認まで行う。
- ChatGPTの画面名や機能名が変わっても、「相談」と「実装」の役割で判断する。
- 同じミスが2回起きた場合は、原因を振り返ってこのルールを更新する。
- 仕様や設計を決める前に、公式の最新情報を優先して確認する。
- 必要に応じて一般ユーザーの優れた事例、GitHub上の実装例、失敗事例も調査する。
- 公式情報、ユーザー事例、推測を区別し、更新日、再現性、安全性を確認する。
- 外部コードや素材は、ライセンス、著作権、個人情報、機密情報、セキュリティを確認してから利用する。
- 調査で得た有効な内容は、採用理由とともに `共有仕様書.md` へ反映する。
- 小さな変更は `共有仕様書.md` へ直接記録する。
- 大きな変更は `docs/specs/` に変更専用仕様書を作り、`共有仕様書.md` から参照する。
- 大きな変更は「仕様確定 → 実装計画 → 作業項目 → 実装・検証」の順で進める。
- 仕様には、目的、変更内容、変更しない範囲、完了条件を記載する。
- 完了時は、実装日、commit、テスト・公開確認結果を変更履歴へ記録する。
